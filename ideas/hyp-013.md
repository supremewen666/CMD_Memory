# Hypothesis 013

## 假设
PrefixGuard 的离线训练 monitor 可以作为 CMD Subagent Judge Monitor 的替代前端，形成 "PrefixGuard 检测 → CMD 归因" 的两层架构，在保持 enum-locked 边界的同时提高异常检测精度。

## 推理过程

PrefixGuard (2605.06455) 在 WebArena/τ²-Bench/SkillsBench/TerminalBench 上达到 0.533-0.900 AUPRC 的在线 failure warning，且 LLM judge 在同一 prefix-warning 协议下显著弱于训练后的 monitor。这表明：

1. CMD 当前使用 rule-based Subagent Judge Monitor（enum-locked `anomaly_reason`）是一种设计选择，但 PrefixGuard 提供了替代方案：用离线 trace 数据训练一个确定性 monitor，输出 risk score 而非枚举原因。

2. PrefixGuard 的 "observability ceiling" 概念可以反过来强化 CMD 的 monitor 边界：当 PrefixGuard 的 risk score 低于理论天花板时，说明 trace 中缺乏可检测信号，此时 CMD 应该跳过 replay（节省成本）；当 risk score 高于天花板时，说明信号充足但 monitor 能力不足，此时应触发 CMD replay。

3. 两层架构保持 CMD 的核心边界不变：PrefixGuard 输出 risk score + detected event type（不作为最终归因），CMD replay 输出 operation-level label + ECS。PrefixGuard 不接触 gold evidence、不写 memory、不产生 ECS。

4. 这个方向不同于 V1 late-stage 的 RPE pre-filter：RPE 基于 evidence-surprise gap，PrefixGuard 基于 trace-to-monitor 的 supervised learning。两者可以互补——RPE 过滤 evidence 层面的 surprise，PrefixGuard 过滤 execution 层面的 anomaly。

## 自评
- 新颖性: 3 — 两层架构本身不新颖，但将 PrefixGuard 的 observability ceiling 用于 CMD 的 replay gating 决策是新的组合
- 可行性: 4 — PrefixGuard 使用监督训练，需要标注 trace 数据；CMD V1 的 mem0 adapter 可以提供这些 trace
- 影响力: 3 — 如果成功，可以将 CMD 的 monitor 从 rule-based 升级为 learned，同时保持 boundary safety；如果失败（PrefixGuard 在 CMD 的 memory-specific traces 上表现不佳），也不影响 CMD 的核心 attribution pipeline

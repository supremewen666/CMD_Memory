# Hypothesis 010

## 假设

CMD-Audit 如果补充真实 retrieval baselines、ranked retrieval trace 和更细的 evidence metrics，将更容易证明自己不是把“弱检索器失败”误当成新方法优势，同时也能更清楚地区分 `retrieval_error` 与 `premature_extraction_error`。

## 推理过程

V0 保持简单是合理的，因为第一阶段需要先验证 package-oracle schema、counterfactual replay、attribution table 和 post-repair replay 能跑通。过早引入 BM25、vector、hybrid/rerank、hard negatives 和完整 ranking metrics，会让第一版工程面过宽。

但如果长期只用 fixture 里的 `retrieved_memory_ids`，CMD 的 retrieval 结论会显得太弱：评审可能质疑所谓 `retrieval_error` 只是因为检索 baseline 不真实，或者所谓 `premature_extraction_error` 会被更强 retriever 解决。因此，V0.5 应加入真实 retriever、ranked trace、Recall@k/MRR/nDCG/Precision@k/context noise ratio，并用 same-entity、temporal conflict、paraphrase、多跳证据、stale memory 和 compression-loss hard negatives 压测。

关键边界是：当 raw-event oracle 能恢复、但 extracted-memory oracle 不能恢复时，即使换更强 retriever，也应优先归因为 `premature_extraction_error`。因为证据没有被保存在 extracted memory 中时，retriever 本身没有可检索对象。

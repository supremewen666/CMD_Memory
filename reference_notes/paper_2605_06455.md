# PrefixGuard: From LLM-Agent Traces to Online Failure-Warning Monitors

arXiv: 2605.06455, 2026-05-07

Authors: Xinmiao Huang, Jinwei Hu, Rajarshi Roy, Changshun Wu, Yi Dong, Xiaowei Huang

## Core Contribution

PrefixGuard is a trace-to-monitor framework that converts raw LLM agent traces into lightweight online failure-warning monitors. It has two stages: (1) **StepView induction** — derives deterministic typed-step adapters from raw trace samples offline; (2) **supervised monitor training** — learns an event abstraction and prefix-risk scorer from terminal outcomes.

## Three Innovations

### 1. StepView: Auto-Induction of Typed Steps from Heterogeneous Traces

Agent traces are highly heterogeneous across benchmarks and frameworks. Traditional approaches hand-author event schemas (brittle) or use LLM judges at deployment time (expensive, unstable). StepView solves this by offline, deterministic induction of typed-step adapters from raw trace samples — automatically recognizing step types and converting unstructured traces into structured event sequences. A single PrefixGuard pipeline works across WebArena, τ²-Bench, SkillsBench, and TerminalBench without per-environment parsers.

### 2. Observability Ceiling: Separating Monitor Error from Missing Signal

A theoretical AUPRC bound that separates monitor error from failures lacking detectable evidence in the observed prefix. Actual data shows 6-49% of failures fall below this ceiling — meaning even a perfect monitor cannot detect them from the prefix alone. This directly answers "when should we give up on online detection and use post-hoc diagnosis?"

### 3. Post-hoc DFA Extraction: Auditable Deterministic Rules

Trained monitors are neural scorers, but PrefixGuard can extract a deterministic finite automaton (DFA) from the trained monitor. The DFA is auditable — state transition rules can be inspected, verified against adversarial samples, and proven immune to prompt injection.

## Key Results

- Best monitors: 0.900/0.710/0.533/0.557 AUPRC (WebArena/τ²-Bench/SkillsBench/TerminalBench)
- +0.137 AUPRC over raw-text controls on average
- LLM judges are substantially weaker under the same prefix-warning protocol
- Observability ceiling reveals 6-49% of failures have no detectable prefix signal

## CMD Relevance

### Architectural Relationship: Complementary, Not Competitive

PrefixGuard and CMD overlap at the online detection layer but diverge completely after detection:

```
PrefixGuard boundary              CMD boundary
     ↓                                ↓
[Trace] → [Monitor] → [risk_score] → END
                        vs
[Memory trace] → [Monitor] → [anomaly] → [Replay] → [Attribution] → [ECS] → [Post-Repair Replay] → [Failure Memory]
```

PrefixGuard's endpoint (risk score) is near CMD's starting point. PrefixGuard can replace CMD's monitor component but cannot replace replay, attribution, ECS, Post-Repair Context Replay, or Failure Memory.

### Detection Signal Source: Different, Complementary

| Dimension | PrefixGuard | CMD Subagent Judge Monitor |
|-----------|------------|---------------------------|
| Signal source | Execution trace (tool calls, action sequences) | Memory state (evidence-answer match, retrieval completeness) |
| What it sees | Agent repeated the same failed tool call 10 times | Evidence was retrieved but answer contradicts evidence |
| What it misses | Reasoning error with clean execution trace | Execution anomaly that doesn't touch memory |
| Training | Supervised on offline traces | Rule-based (no training needed) |
| Output | Risk score (continuous) | anomaly_reason (discrete enum) |
| Determinism | DFA-extractable post-training | Enum-locked by design |
| After anomaly | (not addressed) | Triggers counterfactual replay |

### Validates Three CMD Design Decisions

1. **Rule-based monitor over LLM judge**: PrefixGuard independently finds LLM judges are weak for online warning — exactly why CMD uses enum-locked `anomaly_reason` instead of free-form LLM judgment.
2. **Observability ceiling → replay gating**: When prefix signal is below the ceiling, CMD should skip replay (failure is undetectable from available trace), saving cost for cases where replay budget is better spent elsewhere.
3. **Deterministic detection rules**: Both PrefixGuard (DFA extraction) and CMD (enum lock) favor auditable, deterministic detection over black-box neural scoring.

### Integration Path (hyp-013)

PrefixGuard-style trained monitor can serve as CMD's online detection frontend:
- PrefixGuard fires (risk_score > threshold) → CMD counterfactual replay for attribution
- Observability ceiling gates replay: skip when prefix lacks signal
- DFA extraction maintains CMD's auditability requirement
- Complements (not replaces) RPE pre-filter: PrefixGuard filters on execution trace, RPE filters on evidence surprise

## Open Gap

PrefixGuard detects failure risk but does NOT attribute which operation failed, does NOT generate repair guidance, and does NOT test whether repairs work. CMD's replay engine is the natural next step after PrefixGuard's anomaly flag. The two frameworks are complementary layers in a complete memory debugging stack.
